# 【翻译】在 Codex 中使用 Goals：长时间工作的持久目标

> 来源：OpenAI Cookbook，原文为 `using_goals_in_codex.ipynb`。
> 原文链接：https://github.com/openai/openai-cookbook/blob/main/examples/codex/using_goals_in_codex.ipynb
> 说明：本文为中文翻译，命令名称保留原样，示例中的目标描述翻译为中文以便直接阅读。

Goals 是 Codex 中的持久目标，它能让一个线程跨多轮对话持续朝着明确结果推进。一个 Goal 会给 Codex 一个完成条件：什么状态应该成立，应该如何检查成功，以及哪些约束必须始终保持不变。

Codex 已经很擅长处理边界清晰的编码任务：查看一个仓库、修复 bug、添加测试、解释失败原因，或者实现一个聚焦的改动。Goals 面向的是另一类任务：下一步行动取决于 Codex 在过程中学到了什么。例如性能 profiling、打补丁、跑 benchmark、复现 flaky test，或者把一个研究问题变成有证据支撑的审计报告。

这类任务不需要更长的 prompt，而是需要一个持久目标。有了 Goal，Codex 可以一直看见目标，判断工作是否完成，并在每个中间结果之后选择下一步有用动作，而不需要你反复重述最终目标。

Goal 不是没有边界的后台自治。它是一个有范围、由用户控制的完成契约。你定义结果，Codex 根据线程里的证据工作；Goal 可以被暂停、恢复、清除、完成，也可以因为预算用尽而停止。

## 你将学到什么

读完这篇指南之后，你将能够：

- 判断什么时候 Goal 比一次性 prompt 更合适。
- 写出带有可度量结果、验证面和约束条件的 Goal。
- 使用 `/goal`、`/goal pause`、`/goal resume` 和 `/goal clear` 管理 Goal 的生命周期。

## 前置条件

- 一个支持 Goals 的 Codex 版本。
- 一个有明确终点、并且有证据源可检查的任务，例如测试、benchmark 或最终产物。
- 足够的仓库或研究上下文，让 Codex 能够验证进展，而不是只做口头汇报。

## 快速开始：使用 Goals

当任务有明确终点，但通往终点的路径并不确定时，就适合使用 Goal。

适合的例子包括性能优化、flaky test 调查、依赖迁移、需要复现的 bug 排查、多步骤重构、benchmark 驱动的调优，以及需要交付最终产物的研究任务。对于一次性的简单修改，普通 prompt 仍然是正确工具。

要使用 Goals，先安装或更新 Codex，并确认版本。Goals 从 Codex 0.128.0 开始可用。

使用 npm：

```bash
npm install -g @openai/codex@latest
codex --version
```

使用 Homebrew：

```bash
brew update
brew upgrade --cask codex
codex --version
```

然后使用 `/goal` 加上目标来设置一个 Goal：

```text
/goal 将 p95 延迟降低到 120 ms 以下，同时不让正确性测试回退
```

你可以在同一个命令入口管理生命周期：

```text
/goal          查看当前 Goal
/goal pause    暂停一个活跃的 Goal
/goal resume   恢复一个已暂停的 Goal
/goal clear    移除当前 Goal
```

一旦 Goal 处于活跃状态，Codex 就可以检查代码、运行相关命令、修改文件、测试结果，并持续工作直到触发停止条件。停止条件可能是成功、暂停、清除、中断、预算上限，或者遇到需要用户输入的阻塞。

当你发现自己每一轮都想重复说这些话时，就可以考虑使用 Goals：

```text
继续。
试下一个可能的修复。
再跑一次 benchmark。
现在检查测试。
一直推进，直到这件事真正完成。
```

Goal 会把这层意图明确写出来。

## Goals 和 Prompts 的区别

普通 prompt 的意思是：做接下来这件事。

Goal 的意思是：持续工作，直到这个结果成立。

这个区别很重要。普通请求里，Codex 会执行当前指令，报告结果，然后等待下一步。有了 Goal，线程会拥有一个持久目标。一轮结束后，Codex 可以查看当前证据，并判断目标是否已经满足。如果答案是否定的，并且 Goal 仍然活跃、预算仍然足够，Codex 就可以从最新状态继续推进。

因此，当正确的下一步行动依赖 Codex 刚刚学到的信息时，Goals 最有用。例如：

```text
/goal 在 checkout benchmark 上将 p95 延迟降低到 120 ms 以下，同时保持正确性测试套件通过
```

这不只是一个“提升性能”的请求。它给了 Codex 一个可度量结果、一个验证面和一个约束。Codex 可以运行 benchmark、检查热点路径、做一个定向修改、重新跑 benchmark、运行正确性测试；如果结果仍然不够好，就继续迭代。

实际心智模型很简单：

```text
Prompt：请求 -> 工作 -> 结果 -> 等待

Goal：工作 -> 检查 -> 继续或完成
```

Goal 给 Codex 一条终点线。但工作是否完成，仍然必须接受证据审计。

## 如何写一个 Goal

好的 Goal 不只是更大的 prompt。它是一份紧凑的契约，定义 Codex 应该如何工作、什么算成功，以及如果暂时无法成功应该怎么办。

最强的 Goals 通常会定义六件事：

- 结果：工作完成时什么应该成立。
- 验证面：证明结果的测试、benchmark、报告、产物、命令输出或源材料。
- 约束：Codex 工作过程中哪些东西不能回退。
- 边界：Codex 可以使用哪些文件、工具、数据、仓库或资源。
- 迭代策略：每次尝试之后，Codex 应该如何决定下一步试什么。
- 阻塞停止条件：在当前限制下没有可辩护路径时，Codex 应该何时停止并报告。

一个有用的模板是：

```text
/goal <期望的最终状态>，由 <具体证据> 验证，同时保持 <约束>。使用 <允许的输入、工具或边界>。每次迭代之间，<Codex 如何选择下一步最有价值的行动>。如果被阻塞或没有有效路径，<Codex 应报告什么，以及什么输入可以解锁进展>。
```

比如，下面这个 Goal 可用，但仍然偏薄：

```text
/goal 将 checkout 的 p95 延迟降低到 120 ms 以下，同时不让正确性测试回退
```

更强的版本会给 Codex 更完整的工作契约：

```text
/goal 将 checkout 的 p95 延迟降低到 120 ms 以下，并由 checkout benchmark 验证，同时保持正确性测试套件通过。只使用 checkout 服务、benchmark fixtures 和相关测试。每次迭代之间，记录改了什么、benchmark 显示了什么，以及下一步最值得尝试的实验。如果 benchmark 无法运行或没有有效路径，停止并报告已经尝试的路径、已收集的证据、阻塞点，以及继续推进所需的下一个输入。
```

对研究和调查任务，同样适用这个原则。尤其当精确证明可能不可得时，要在开始前定义证据标准：

```text
/goal 使用可获得的论文材料和本地资源，产出最强的、有证据支撑的论文复现。尽可能尝试主要结果，在可行处验证输出，最后交付一份报告，区分已确认发现、近似重建、被阻塞的主张和剩余不确定性。
```

这种 Goal 给了 Codex 调查空间，同时让最终结果保持诚实。它不只是说“继续做”，而是说明“完成”、“阻塞”和“仍有不确定性”分别意味着什么。

当任务清楚但 Goal 还不清楚时，Codex 也可以帮助你写 Goal。一个简单的两步流程很有效：首先，用普通语言描述工作，并让 Codex 把它改写成一个 Goal 草稿；然后，你审阅草稿，收紧成功条件、验证面、约束和阻塞停止条件，最后再激活它。

例如：

```text
帮我把这件事改写成一个强 `/goal`：我希望 Codex 持续处理这个 flaky 的 checkout 测试，直到我们要么用证据修好它，要么能清楚解释是什么阻塞了进展。
```

Codex 可以据此提出一个更完整的 Goal，只询问真正必要的缺失信息，并给你一个更干净的 `/goal` 命令。

## Goal 活跃后会发生什么变化

当一个 Goal 处于活跃状态时，有三件事会变化。

第一，目标始终可见。如果 Codex 运行测试并失败，线程仍然保留原始目标。如果 benchmark 有改善但还没达到阈值，Codex 可以继续。如果一条研究路径遇到缺失数据，Codex 可以调整证据计划，而不会忘记研究标准。

第二，线程空闲时可以继续推进。Codex 不会在另一个 turn 仍在运行、用户输入正在排队、或者线程中还有其他待处理工作时继续。只有当线程空闲、Goal 活跃且预算充足时，它才会继续。

第三，完成必须基于证据。Goal 不应该因为模型“觉得差不多完成了”就被标记为完成。只有当目标已经被相关文件、测试、日志、benchmark 输出、生成产物或其他具体证据检查过之后，才应该完成。

这就是设计中心：Codex 可以持续推进，但由证据决定它是否完成。

## Codex 中 Goals 的设计方式

Goals 被实现为持久化的线程状态，而不是全局记忆，也不是项目级指令。这个设计选择很重要：目标属于包含相关上下文的线程，包括 Codex 查看过的文件、运行过的命令、产生的 diff、看过的日志，以及一路建立起来的推理轨迹。

![图 2：Goal 为当前线程加入持久状态、继续推进能力、控制入口和证据检查](https://files.mdnice.com/user/59/875a58a8-1948-4e89-ab65-453233d7b608.png)

*图 2. Goal 为当前线程加入持久状态、继续推进能力、控制入口和证据检查。*

在架构层面，Goal 是一个持久的、线程作用域内的状态。它记录目标、生命周期、预算和进度账本，供 Codex 随时间评估线程。关键边界是作用域：Goal 属于当前线程，而不是全局记忆或项目指令。

Codex 会把这个状态视为用户、模型和线程之间的契约。Goal 可以处于活跃、暂停、完成或预算受限状态。这些状态决定 Codex 是否可以继续、是否应该等待用户，以及是否应该总结进展而不是开始新工作。

继续推进是事件驱动的，而不是简单循环。Codex 只会在安全边界检查是否继续：一轮工作已经结束，没有其他工作待处理，没有用户输入排队，并且线程处于空闲状态。

调度器的行为是刻意保守的。只有计划、没有实际执行的工作不会触发继续。中断会暂停目标。在合适的时候，恢复线程可以恢复目标。如果某次自动继续没有调用任何工具，那么下一次自动继续会被抑制，避免 Codex 原地空转。

![图 3：只有当 Goal 活跃、线程空闲且没有用户输入排队时，Codex 才会继续](https://files.mdnice.com/user/59/4ffc7f52-9892-4448-9244-4c5d3109de92.png)

*图 3. 只有当 Goal 活跃、线程空闲且没有用户输入排队时，Codex 才会继续。*

Prompt 层也强化了同一套架构。继续推进的 prompt 会让 Codex 围绕活跃目标工作，但也要求在完成前做一次审计。Codex 必须把目标和具体证据对照：改过的文件、运行过的命令、通过的测试、benchmark 输出、生成产物或研究证据。

预算处理是显式的。当预算达到上限时，Codex 应该停止实质性工作，总结进展和阻塞点，并指出下一步有用动作。达到预算上限不等于完成目标。

工具契约会限制生命周期权限。模型可以启动一个 Goal，也可以在证据支持完成时标记已有 Goal 完成。暂停、恢复、清除和预算受限转换仍然由用户或系统控制。

你需要记住的架构图景是：Goal 是一个线程作用域内的完成契约。它组合了持久目标状态、生命周期控制、继续推进策略、预算记账和基于证据的完成判定。它的目的不是让 Codex 永远循环，而是让目标一直存在，直到证据说明工作已经完成。

## 把弱 Goal 改成强 Goal

弱：

```text
/goal 提升性能
```

强：

```text
/goal 在 checkout benchmark 上将 p95 延迟降低到 120 ms 以下，同时保持正确性测试套件通过
```

### 示例：性能调优

![图 4：强 Goal 会明确最终状态、验证面和约束](https://files.mdnice.com/user/59/d2acf1cc-e8bf-4dff-88dd-e8ea980c1a35.png)

*图 4. 强 Goal 会明确最终状态、验证面和约束。*

更强的版本给了 Codex 三样东西：结果、验证方法和约束。它也告诉 Codex 什么时候不应该停止。如果 p95 从 180 ms 改善到 135 ms，Goal 还没完成。如果延迟降到 120 ms 以下但正确性测试失败，Goal 也没完成。如果 benchmark 无法运行，Codex 必须报告这个阻塞，而不是宣称成功。

这条规则也适用于性能以外的工作：Goal 应该足够具体，能够被验证；同时也要足够开放，支持探索。

Goal 应该窄到可以审计，也应该宽到能让 Codex 选择下一步动作。“修复失败的 checkout 测试”可能太窄，因为真正问题可能来自上游依赖。“改进整个系统”又太宽，因为没有审计面。“在当前分支上让 checkout 测试套件通过，同时不改变公共 API 行为”会好得多。

同样原则也适用于生成产物。一个弱 Goal 会说：

```text
/goal 为这个功能写文档
```

更强的 Goal 会说：

```text
/goal 产出一页关于 Goals 的文档，解释生命周期、命令入口和两个示例。验证这页文档可以在本地构建，并确认所有引用命令都符合当前 CLI 行为。
```

第二个 Goal 给了 Codex 可检查的对象：一页文档、一个构建命令，以及命令行为。

这种纪律对研究型 Goals 更重要。调查开始前就要定义证据标准：什么算精确复现，什么算部分重建，什么算代理支持，什么应该被视为阻塞。

强研究 Goal 应该要求 Codex 建立主张清单，把主张映射到证据，实现可行部分，标记阻塞点，并产出一份审计报告，区分已确认主张、仅能支持的证据、被阻塞主张和剩余不确定性。

这样可以让 Goal 足够窄，能够审计；同时不把完整路径预先写死。Codex 可以选择下一步动作，但完成标准是固定的。

## 用 Goals 处理复杂研究：复现一篇量化论文

下面是一个具体的研究型 Goal 示例，展示这些原则如何落地。

案例研究来自 Buehler、Gonon、Teichmann 和 Wood 的论文《Deep Hedging》。这篇论文的问题是：在不同风险偏好、交易成本和高维市场设置下，神经交易策略能否复现基于模型的对冲。正确的 Goal 不是抽象地“复现论文”，而是尝试论文中的主要数值主张，把精确机制和近似训练替代品区分开，并明确说明哪些内容无法从可获得材料中精确重放。

一个弱研究 Goal 可能是：

```text
/goal 复现 Buehler 等人的《Deep Hedging》
```

这太欠定义。它没有说明哪个章节重要，什么算复现，如何处理不可得的训练状态，也没有说明如何区分接近的数值匹配和精确重放。

更好的 Goal 是：

```text
/goal 使用可获得的论文材料和本地资源，产出对 Buehler 等人《Deep Hedging》最强的、有证据支撑的复现。尝试每个主要结果，验证输出，并最终交付一份报告，区分已复现机制、近似训练结果、被阻塞的精确重放和剩余不确定性。
```

更强的版本之所以有效，是因为它命名了证据标准和最终产物。Codex 不只是要产出一个看起来很厉害的复现，而是要尽量减少不确定性，同时不夸大现有证据能支持的范围。

![图 5：研究型 Goal 会先把论文拆成不同证据通道，再声明状态](https://files.mdnice.com/user/59/588580bb-1259-4dbf-9f50-a0c66b2a7875.png)

*图 5. 研究型 Goal 会先把论文拆成不同证据通道，再声明状态。*

在实践中，这个 Goal 给调查过程提供了一个具体的操作契约。

Codex 用它来：

- 区分主要主张和辅助主张。
- 将这些主张映射到可获得证据。
- 重建可以在本地测试的部分。
- 标记无法从现有材料中精确复现的主张。

有几部分是可行的。Codex 重建了定价和对冲机制，复现了 Heston 参考价格，为 CVaR 对冲实验训练了策略，重建了主要直方图和对冲曲面产物，复现了 Black-Scholes 交易成本斜率，并对 Heston 交易成本和高维示例运行了训练后检查。

有些主张因为缺少源材料仍然被阻塞。论文没有提供精确随机种子、生成的训练路径、TensorFlow graph、优化器状态、checkpoint 或完整原始仿真状态。这意味着最诚实的最强结果是部分且近似的复现，而不是精确的神经网络重放。

这就是 Goal 的价值。它让工作在遇到阻塞后仍然继续推进，同时也让最终表述保持诚实。一个训练得到的替代模型可以支持某个主张，一个接近的数值匹配可以提高置信度，一个重建图像可以验证部分结果，但这些都不应该被描述成精确恢复了原始实验。

![图 6：最终输出应该保留不同层级的认知支持](https://files.mdnice.com/user/59/61c099aa-d516-4635-91e2-83e6b940906a.png)

*图 6. 最终输出应该保留不同层级的认知支持。*

最终报告应该保留这些不同层级的支持，而不是把它们压平成一个简单的成功声明。

例如，一条审计账本可以写成这样：

```text
主张：Deep hedging 能在无交易成本时近似完全市场下的 Heston 对冲。
路径：重建模型机制、参考对冲比较，并训练神经策略。
证据面：价格检查、直方图和对冲曲面。
状态：接近的近似复现。
剩余不确定性：原始训练路径、随机种子和 checkpoint 不可获得。
```

这就是 Goals 在研究任务中的演示价值。它们让 Codex 能够在模糊性中继续工作，同时防止一个看起来合理的产物变成过度声称的结论。Goal 不只是要求 Codex 完成；它定义了“完成”的含义：逐条主张的审计，基于证据，明确近似性，并诚实标出复现和重放之间的边界。

## 什么时候不要使用 Goals

Goals 并不适合所有任务。

不要为一行代码修改、简单解释、短代码审查，或者你只想要一个答案然后停止的问题使用 Goal。普通 Codex prompt 更适合这些情况。

当终点模糊时，也不要使用 Goal。“让它变得更好”没有给 Codex 可靠的完成条件。“重构这段代码”也很弱，除非你定义了期望的最终状态、测试和约束。

不要用 Goal 掩盖不确定性。如果数据可能不可得，就在 Goal 里说明。如果 benchmark 可能不稳定，就说明应该如何处理。如果允许代理证据，就定义它应该如何被标记。

当任务同时具备三个属性时，Goals 最强：持久目标、基于证据的终点，以及可能需要多轮调查的路径。

## 结论：让目标持续存在，但让证据决定结果

Goals 改变了 Codex 的工作模型。它把一个线程从一串相互孤立的 prompts，变成围绕明确结果展开的有状态工作循环。

这套架构有意保持边界。Goal 的作用域是线程，携带生命周期状态和预算记账，可以暂停、恢复、清除、完成，也可以因为预算用尽而停止。Codex 可以持续推进，但只能在用户定义的契约内行动。

这让 Goals 适合 Codex 本来就最有价值的工作：debug、优化、迁移、测试和研究。用户提供目标，Codex 跟随证据。Goal 把两者连接起来，直到工作真正完成，或者诚实地被阻塞。

对于复杂研究来说，这就是“生成一个答案”和“产出一份审计”的区别。好的 Goal 不只是要求 Codex 完成。它告诉 Codex，什么才算完成。
